# inoue-movie4 — プロジェクト計画書（改訂版）

---

## 勝ち筋（一言）

> **「AIが選んだ商品を、HOOKで止める動画で刈り取る」**

- 上流（何を売るか）: Creative Centerのデータを元にAIが商品候補を提案。人間が承認。
- 下流（どう見せるか）: 最初の3秒（HOOK）を磨くことがCVRの主変数。
- 物量: 1商品につき10本生成 × 10アカウントに1本ずつ投稿。1アカウントあたり1日1本でアルゴリズムに最適化しつつ、全体では圧倒的な量とA/Bデータを確保する。

---

## 目的

**TikTok Shop で月間 GMV 合計 5,000 万円を達成する。**
1アカウント1日1投稿。10アカウント並列運用。1動画あたりの購買数を最大化する。
アカウントあたりの内訳（例: 5アカウント×1,000万円 or 10アカウント×500万円）はPhase 4のデータを見て決定する。

### 根拠の構造

1. **実績参照（保守的設定）**
井上式が 1,500 万円ならその 67% = 1,000 万円。ブランド立ち上げ期・AI 生成の品質差を加味した保守係数。「届かないかも」ではなく「確実に届く最低ライン」として置く。

2. **収益構造**
自動化 1 アカウントで 1,000 万円 → 純利益 100 万円。アカウント数を増やすことで線形にスケール。APIコストの増分は微小（アカウント追加あたり +3,000 円/月程度）。

3. **1人運用の物理上限**
動画生成は自動化できるが、在庫管理・CS は人力。1,000 万円が「1日 2〜3 時間で持続できる上限」という意味で自然な目標になる。

4. **損益分岐がほぼゼロ**
月 1.1 万円売れれば黒字。1,000 万円は損益分岐の 1,000 倍。

5. **スケールの非線形性**
1,000 万 → 2,000 万にするとき、APIコストの増分は約 3,000 円。これが自動化の本質的な強み。

## GMV 逆算

```
目標: 月間 GMV 合計 5,000 万円

平均単価 3,000 円 × 16,667 件/月の購入
  → 1日 556 件の購入が必要（10アカウント合計）
  → 1アカウントあたり 56 件/日

CVR 0.5% で → 1アカウントあたり 11,200 imp/日 必要
1投稿/日 → 1本で 11,200 imp を稼ぐ

【結論】CVRが全て。HOOKで止めて、刈り取る動画の質が勝負。
CVR 0.5% → 1.0% になれば、同じ投稿数で GMV 2倍。
```

## パイプライン（決定版）

### 基本思想

```
00: AIが商品候補をスカウト → 人間が承認 → DBに記録
    ↓
商品画像 1 枚
  → [01] 商品を解析して訴求ポイントを言語化
  → [02] 購買ファネル × HOOKバリエーション 10本分のカット構成を設計
           （過去DBの勝ちパターンを参照）
  → [03] 各カットの画角画像を AI 生成（Flux.1）← 並列
  → [04] 各画像から AI 動画クリップを生成 ← 並列
  → [05] クリップを結合・字幕・AI音声・BGM を追加
  → [06] QA チェック
  → [07] 投稿文・ハッシュタグ生成
  → [08] CVRをDBに記録 → 勝ちパターンを更新
  → 完成（10本 × 20〜25秒・4K縦型）
```

### クリップ設計（5〜8本 × 3〜5秒 = 20〜25秒）

| # | 役割 | 画角 | モーション | 秒数 |
|---|------|------|-----------|------|
| 0 | HOOK | wide（全体引き） | slow dolly in | 5秒 |
| 1 | BENEFIT | close（特徴部） | gentle rotation | 5秒 |
| 2 | BENEFIT | angle（斜め45度） | subtle float | 3秒 |
| 3 | PROOF | scene（使用シーン） | natural movement | 5秒 |
| 4 | CTA | front（正面フラット） | static / slow orbit | 4秒 |

HOOKは10本それぞれで異なるバリエーションを生成し、A/Bデータを1日で取得する。

### 技術スタック

| 工程 | API / ツール | 単価 | 理由 |
|------|------------|------|------|
| 商品スカウト | Claude API + Creative Center | ~$0.02 | カテゴリトレンド分析・候補生成 |
| 商品解析 | Claude Vision | ~$0.01 | 画像を読んで特徴を言語化 |
| 台本生成 | Claude API | ~$0.05 | ファネル構成・DBの勝ちパターン参照 |
| 画角画像生成 | Flux.1 | $0.02/枚 | 元画像 reference で一貫性確保 |
| AI動画生成 | config.js で切替可能（デフォルト: Kling 3.0） | 600円/本（20秒） | 差し替え容易な設計 |
| 音声 | ElevenLabs（日本語） | 100円/本 | 参考動画品質 |
| BGM | ロイヤリティフリー素材 | 無料 | ナレーション -20dB でミックス |
| 動画編集 | ffmpeg | 無料 | 結合・字幕・エンコード |
| DB | SQLite（後にPostgreSQL移行可） | 無料 | 勝ちパターン・CVR蓄積 |

**1本あたりコスト: 約700円（Kling 600円 + ElevenLabs 100円 + その他）/ 100本/日・月3,000本: 約210万円/月**

### 生成時間（並列実行時・10本）

```
商品解析・台本生成（10本分）:    2〜3分
画角画像生成（50枚並列）:        3〜4分
AI動画生成（50本並列）:         5〜8分
編集・音声・BGM（10本並列）:    3〜4分
─────────────────────────────
合計: 約 13〜19分 / 10本
```

## データ設計（永続化）

/tmp に捨てていたデータをDBに残す。これが「時間が経つほど強くなる」の基盤。

### 保存するデータ

| テーブル | 内容 | 用途 |
|---------|------|------|
| `jobs` | jobId・商品・生成パラメータ・タイムスタンプ | 全ジョブの記録 |
| `shots` | カット構成・HOOKの型・尺・テキスト | 構成パターンの蓄積 |
| `metrics` | CVR・3秒維持率・完視聴率・インプ数 | 勝ちパターンの判定 |
| `products` | 商品候補・承認/却下・承認者コメント | 商品選定の学習 |
| `patterns` | 採択済み勝ちパターン（shot_idと指標を紐付け） | shot-plannerの参照元 |

### 勝ちパターンの自動更新フロー

```
7日後に metrics を集計
  → CVR ≥ baseline × 1.15 かつ 3,000 imp 以上
  → patterns テーブルに採択
  → 次回の shot-planner がこのパターンを優先参照
```

## モデル差し替え設計

`config.js` の1行を変えるだけで動画生成APIを切り替えられる。

```javascript
module.exports = {
  VIDEO_GEN_PROVIDER: "kling",  // "runway" | "sora" | "kling" に変更可
  IMAGE_GEN_PROVIDER: "flux1",  // "dalle" | "midjourney" に変更可
  TTS_PROVIDER: "elevenlabs",   // "voicevox" | "nijivoice" に変更可
};
```

半年後により良いAPIが出ても、データはそのまま新モデルに食わせられる。

## フェーズ計画

### Phase 1 ✅ 完了 — 仕様・設計の固め

### Phase 2 ← **今ここ** — E2E プロトタイプ

**作業順:**
1. `db/schema.sql` を作成（最初に）← データ設計を後回しにしない
2. `agents/00_product-scout/` を作成 → Creative Center分析確認
3. `agents/01_analyze/` を作成 → Claude Vision で動作確認
4. `agents/02_shot-planner/` を作成 → DB参照込みで台本生成確認
5. `agents/03_image-gen/` → Flux.1 API で画角画像生成確認
6. `agents/04_video-gen/` → config.js 経由で動画生成確認
7. `agents/05_assembly/` → 結合・音声・字幕確認
8. `agents/06_qa/` `07_publish-prep/` `08_measurement/` を作成
9. `cli.js` + `orchestrator.js` で E2E 接続（10本並列）

**合格条件:** `node cli.js generate product.jpg --title "爪切り"` で final.mp4 × 10本が出力され、DBにジョブが記録される

### Phase 3 — 品質を参考動画レベルに
- HOOKバリエーション10パターンの質を目視検証
- ElevenLabs 日本語音声の品質チューニング
- 「手あり」プロンプトの精度検証
- 字幕フォント・タイミング調整

**合格条件:** 参考動画と並べて見劣りしない

### Phase 4 — CVR 計測ループ稼働
- 10アカウントで1日10本投稿を開始
- TikTok Analytics CSV を `node cli.js measure` で取り込み → DBに書き込み
- 7日間 × 3,000 imp 以上で採択判定（CVR ≥ baseline × 1.15）
- 採択パターンを patterns テーブルに記録 → shot-planner が自動参照

### Phase 5 — CVR 施策の A/B テスト

| 優先度 | 施策 | CVR 仮説 |
|--------|------|---------|
| 1 | HOOKバリエーション（現行10パターン） | ベースライン確立・最重要 |
| 2 | 手あり vs 手なし | 使用感のリアリティ効果 |
| 3 | 3カット vs 5カット vs 8カット | 最適尺の特定 |
| 4 | レビュー数字を PROOF に強調 | 社会的証明 |
| 5 | CTA に価格を大きく表示 | 購買決断の後押し |
| 6 | 比較型（Before/After） | 差分の可視化 |
| 7 | HOOK に価格を出す | 価格訴求の前倒し |
| 8 | UGC 風テイスト | 広告っぽくない信頼感 |
| 9 | ペルソナ別テキスト | ターゲット精度 |
| 10 | 季節・イベント連動 | 需要タイミング |
