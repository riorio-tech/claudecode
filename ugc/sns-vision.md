# SNS 自動運用システム — 上段設計

> このドキュメントは `ugc/` と `sns-manage/` 両プロジェクトの北極星。  
> 実装判断に迷ったとき、必ずここに立ち返る。

---

## このシステムは何か

「動画を作るツール」でも「投稿するBOT」でもない。  
**SNS を自律的に運用し、使い続けるほど賢くなる永続的な資産**。

商品が入れば動画が出て投稿され、結果が次の動画に反映される。  
投稿するたびに知識が蓄積され、失敗するたびにシステムが強化される。  
人間は商品と目標を与えるだけでいい。

---

## このシステムが創る価値

| 観点 | Before（従来のSNS運用） | After（このシステム） |
|---|---|---|
| 投稿工数 | 担当者が週10h以上かける | 週2h以下（商品入稿のみ） |
| コンテンツ品質 | 担当者スキル依存・バラつき大 | プロンプト設計済みの一定品質 |
| 改善サイクル | 「なんとなく」の感覚改善 | 数値 → 原因分類 → 次回スクリプトに反映 |
| 知識の帰属 | 担当者が退職すると消える | DB に永続化・チームの資産になる |
| 失敗の活用 | 失敗投稿は「なかったこと」 | `failure_patterns` に分類・次回企画に注入 |

**ROI 目安**: 週8h削減 × 時給3,000円 = **月10万円相当の工数削減**

### 解決しないこと（スコープ外）

- 商品力・価格競争力（売れるかはシステムではなく商品が決める）
- ブランド戦略・ポジショニング（インプットとして必要）
- フォロワーの購買転換率（LP・商品ページの管轄外）

---

## ユーザーが求める 3 つの本質ニーズ

SNS 運用代行を依頼・導入する会社が本質的に求めていること：

| # | ニーズ | 本音 |
|---|---|---|
| 1 | **リソース不足の解消（時間短縮）** | 投稿・返信・分析に使う人的リソースがない。担当者の工数を削りたい |
| 2 | **専門ノウハウによる成果** | フォロワーが増えない・集客につながらない。何が正解か分からない |
| 3 | **ブランドに合った質の高い投稿** | 素人っぽい動画や文章はブランドイメージを壊す。プロ品質を維持したい |

**特に重視される 3 要素:**

```
① 見栄えの良いコンテンツ作成
   → 動画の画質・構成・テキストが "プロが作った" と感じられるクオリティ

② 客観的な分析・改善提案
   → 感覚ではなくデータで「どれが効いたか」「次は何を変えるべきか」を提示

③ 迅速なファン対応
   → コメントへの返信が遅いとファン離れにつながる。24時間以内の対応が理想
```

## 設計哲学（5 つ）

### I. Content as Code
スクリプト・フック・CTA はコードとして管理する。  
何が効いたかが Git に残り、再現・改善・ロールバックができる。  
「感覚の属人化」をなくし、チームの知識資産にする。

### II. 閉じたフィードバックループ
投稿 → 効果測定 → スクリプト改善 → 再投稿。  
毎サイクルで精度が上がる。外部ツールに依存しない自律改善エンジン。

### III. プロバイダ非依存
HeyGen が使えなくなっても MakeUGC に戻せる。  
TikTok が仕様変更しても他プラットフォームに逃げられる。  
特定サービスへの依存はリスク。インターフェースで吸収する。

### IV. アンチフラジリティ（Nassim Nicholas Taleb）
> 「壊れやすいものはストレスで崩れ、頑強なものは耐える。しかしアンチフラジャイルなものはストレスから栄養を得て、強くなる。」

従来のSNSツールは「勝ちパターン」だけを学習する。しかし最も情報量が多いのは「バズらなかった投稿」だ。  
失敗を記録しないシステムは、成功の条件を本当に理解していない。

- `patterns`（勝ち）と `failure_patterns`（負け）を対称のテーブルとして持つ
- 失敗モードを分類・記録し、次の企画に「やってはいけないこと」として注入
- `knowledge_base` が実験のたびに `confidence` を積み上げる

### V. 模倣欲望（René Girard）
> 「欲望はオリジナルではない。人は他者の欲望を模倣することで自分の欲望を形成する。」

TikTok のバイラル動画は「良い商品を紹介している」のではなく、「強烈に欲しがっている人間を見せている」から伝播する。

- コンテンツは商品主語でなく「欲しがっている人間の状態・感情・場面」を主語に書く
- `desire_centric` vs `object_centric` の A/B を毎回生成し、どちらが勝つかを学習
- `desire_map` テーブルに欲望連鎖を蓄積する

#### 2つの哲学が交差する場所
**アンチフラジリティ × 模倣欲望 = 「失敗した欲望の分析」**

失敗の要因を模倣欲望フレームで分類する:
- `wrong_mediator` — 欲望の媒介者の設定が読者に近くなかった
- `wrong_emotion` — 感情の種類が間違っていた（憧れの場面で不安を刺激した）
- `object_centric_drift` — コンテンツが商品主語に戻ってしまった
- `timing` — タイミング・文脈が合わなかった

## 成果の定義（3 レイヤー）

「成果」を3つのレイヤーで定義する。計測できないものは改善できない。

### Layer 1: 北極星指標（最終的に測りたいもの）

| 指標 | 目標値 |
|---|---|
| **商品購入数 / 月** | ベースライン比 +15%/月 |
| **新規顧客獲得単価（CAC）** | 既存施策より -20% 以下 |

### Layer 2: SNS 先行指標（毎日追う）

| 指標 | 合格ライン | 改善トリガー |
|---|---|---|
| 完視聴率 | ≥ 30% | < 20% でフック見直し |
| CTR（リンククリック率） | ≥ 2% | < 1% で CTA 見直し |
| 保存率 | ≥ 3% | < 1% でスクリプト見直し |
| フック離脱率 | < 50% | ≥ 60% でフック差し替え |

### Layer 3: システム運用指標（システム自体の健全性）

| 指標 | 目標値 |
|---|---|
| 投稿継続率 | ≥ 90%（月30投稿なら27本以上） |
| 動画生成コスト | ≤ 200円/本 |
| 担当者工数 | ≤ 2時間/週（導入前比 -80%） |
| パイプライン成功率 | ≥ 95% |

### 永続資産の成長指標（sns-manage）

| 指標 | 目標値 |
|---|---|
| knowledge_base confidence 平均 | 0.7 以上（3ヶ月後） |
| failure_patterns 蓄積数 | 20件以上（1ヶ月後） |
| experiment_log supportsDesireTheory 率 | 60% 以上 |

### 「成果なし」の判断基準

以下の状態が **4週間継続** した場合、戦略を根本から見直す:
- 完視聴率 < 20% かつ フック離脱率 ≥ 60%
- CTR < 0.5%
- 投稿継続率 < 80%
- knowledge_base confidence が3ヶ月後も 0.5 未満

## サブシステム構成

```
[ SNS 自動運用システム ]
        │
        ├─ sns-manage/    ← テキスト投稿・分析・学習エンジン
        │   ├─ Twitter / Threads / TikTok / Instagram / YouTube
        │   ├─ 01_research → 02_planning → 03_writer → 06_publish
        │   │                              → 07_analytics → 08_evaluate
        │   └─ 永続資産 DB（patterns / failure_patterns / knowledge_base / desire_map）
        │
        └─ ugc/           ← 動画コンテンツ生成エンジン
            ├─ 商品画像 → HeyGen動画 → アセンブリ → SNS配信 → analytics
            └─ 将来: sns-manage の 04_design agent および 06_publish と接続予定
```

**責任の分担:**

| 関心 | sns-manage | ugc |
|---|---|---|
| テキスト投稿（Twitter/Threads） | ✅ | — |
| 動画生成（HeyGen/MakeUGC） | — | ✅ |
| SNS投稿（TikTok/Instagram/YouTube） | Phase 2 実装中 | Phase 2 実装中 |
| 効果測定・分析 | ✅ | Phase 3 予定 |
| 知識・パターン蓄積 | ✅ | Phase 3 で連携予定 |

```
   09_report: daily_snapshots → Google Sheets + 週次 AI レポート
```

## Claude Code × UGC ベストプラクティス

`ugc/pipeline/` の実装から抽出した実証済みパターン。このシステムを拡張するときの設計指針として使う。

### a. JSON-Only 出力制約

プロンプト末尾に必須: `"Return ONLY valid JSON (no markdown, no explanation)"`

- マークダウンラッピングを防ぎ、応答から `/\{[\s\S]*\}/` で抽出するだけでパース完了
- 失敗時は `"stage-name: no JSON in Claude response:\n{先頭300字}"` で即 throw — 闇雲にリトライしない

### b. ロール → コンテキスト → タスク → スキーマ（全ステージ共通）

```
1. "You are a [役割]..."
2. [前段階の出力をJSONで投入]
3. 制約・ルール列挙
4. 出力スキーマ（キー・型・制約値）
5. "Return ONLY valid JSON"
```

全ステージ同一構造にするとデバッグが容易になる。プロンプト差分だけでステージの違いが分かる。

### c. ステージ間データは JSON ファイルで繋ぐ

`01_analyze.json` → `02_research.json` → `03_script-plan.json` の形でジョブディレクトリに保存。  
パイプラインが中断しても `resume.js` でファイルを読み込んで途中から再実行できる。

### d. config.js ゲッターパターン（ES module の罠）

```javascript
// NG: import 時に解決される（dotenv.config() より先に評価される）
export const MODEL = process.env.CLAUDE_MODEL ?? 'claude-sonnet-4-6';

// OK: 呼び出し時に解決される
export const config = {
  get CLAUDE_MODEL() { return process.env.CLAUDE_MODEL ?? 'claude-sonnet-4-6'; }
};
```

ES module は `import` がホイスティングされるため、静的定数では `.env` が未ロードの状態になる。

### e. Claude を QA ステージとして組み込む

- **Stage 6 `code-review.js`**: パイプライン自身のソースコードを Claude にレビューさせる（自己点検）
- **`eval.js`**: 生成動画を Claude Vision でスコアリングし、閾値未満なら再生成トリガー
- 生成 → 自動評価 → 判定 → 再生成 のループが完全自律化できる

### f. ffmpeg × 日本語テキスト

`drawtext` フィルタで日本語を使う場合は `textfile=` 経由でテンポラリファイルに書き出す。  
フィルタ文字列内のアポストロフィ（`d'Alba`）・コロンのエスケープ問題を完全回避できる。  
`fontfile=` を明示しないと ffmpeg 4.x 以前で `No font filename provided` エラー。
